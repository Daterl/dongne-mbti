# README 리라이트 — 동네 MBTI

## Problem Frame

현재 README는 기술 스택 나열 중심이다. "이 앱이 누구의 어떤 고민을 해결하는가"가 없어서
처음 방문한 사람이 앱의 가치를 직관적으로 파악하기 어렵다.

좋은 README는 **문제 → 사람 → 해결책 → 기술** 순서로 흘러야 한다.

---

## 요구사항

**구조 & 흐름**
- R1. 헤더에 프로젝트명 + 한 줄 가치 제안 + 뱃지(Snowflake, Python) 포함
- R2. 기존 서비스(직방·다방·호갱노노·카카오맵)의 한계를 1-2줄로 간결하게 서술 — 차별점은 억지 주장이 아닌 기능으로 보여줌
- R3. 사용자 페르소나 3개를 스토리 형식으로 제시 → 각자의 페인포인트가 어떤 탭에서 해소되는지 연결
- R4. 핵심 기능 3탭을 사용자 입장 언어로 설명 (기술 용어 최소화)
- R5. 기술 스택·데이터 소스·아키텍처는 별도 섹션으로 하단에 배치
- R6. Cortex AI 6개 기능 목록은 "어디에 쓰이는가" 맥락과 함께 서술 (기능 나열 아님)

**페르소나 정의 (3개)**
- R7. 페르소나 A — 이사 예정인 30대 직장인: "직방에서 가격은 알겠는데 이 동네가 나랑 맞는지 모르겠다" → Tab 1 MBTI 카드
- R8. 페르소나 B — 자녀 학군 고민 중인 40대 부모: "조건에 맞는 동네를 하나하나 찾기 힘들다" → Tab 2 자연어 추천
- R9. 페르소나 C — 전세 갱신 vs 이사 타이밍 고민 중인 신혼부부: "지금 이사해도 되는 타이밍인지 모르겠다" → Tab 3 이사 예보

**기술 섹션**
- R10. 데이터 소스 4종(SPH·RICHGO·Telecom + Snowflake Marketplace)을 각각 "무엇을 알 수 있는 데이터인가" 한 줄 설명과 함께 기재
- R11. Snowflake Cortex AI 기능 6개를 앱 내 사용 위치와 함께 표로 정리
- R12. 아키텍처 다이어그램 유지 (Mermaid)
- R13. 분석 범위(서울 3구 55개 동) + 데이터 피봇 배경을 간략히 설명 — 25구 전체 대신 3구 딥다이브를 선택한 이유 포함

**제외**
- R14. 데모 전달력·발표 관련 내용은 README에 포함하지 않음 (데모 영상으로 증명)
- R15. 설치 가이드, 로컬 실행 방법 불필요 — Streamlit in Snowflake로 배포되어 코드 실행 불필요

## Success Criteria

- 처음 방문한 사람이 스크롤 한 번에 "이 앱이 나에게 필요한가"를 판단할 수 있다
- 기술 스택보다 사용자 가치가 먼저 눈에 들어온다
- 페르소나 중 하나에 공감되면 자연스럽게 아래로 읽고 싶어진다

## Key Decisions

- **문제 먼저, 기술 나중**: 심사위원을 직접 타깃하지 않아도 "왜 Snowflake가 필요했는가"가 자연스럽게 드러나도록 구성
- **페르소나 형식**: 카드형 테이블보다 짧은 스토리 문장이 공감을 더 잘 끌어냄
- **경쟁 서비스 비교**: 공격적 비교 대신 "기존 서비스가 다루지 않는 영역"으로 포지셔닝
